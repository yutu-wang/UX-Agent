## 必改清單（交付前需完成）

依影響力 × 修復成本排序，非純粹嚴重度排序。

---

### M1｜Wizard 缺乏「略過試算」逃生口
**來源：** H3 Finding 2
**影響：** 中等 — 已有初步了解、只想確認特定資訊的用戶（多為回訪用戶）被迫走完三步，或只能使用 nav 連結跳離。這在重複訪問的情境下摩擦感最強。
**修復：**
在 wizard header `w-header-title`「找出適合你的方案」右側加低調連結：
```html
<a href="#plans" style="font-size:.8125rem;color:var(--muted);font-weight:500;text-decoration:none;">
  直接比較 1.0 / 2.0 →
</a>
```
連結錨點指向頁面內「如何運作」區塊或獨立的方案比較表。
**成本：** 低（一行 HTML + 錨點）

---

### M2｜結果卡未說明推薦邏輯（為什麼不是 2.0）
**來源：** H6
**影響：** 中等 — 用戶看到「推薦 1.0」但不知道推薦依據。研究中 P4 明確說「我不知道 2.0 的好處具體在哪裡」，若結果頁也不解釋，決策信心不足。目前的 callout（「你的路線用 1.0 就夠」）是正確方向，但缺少可量化的比較。
**修復：**
在費用明細 grid 下方、申辦按鈕上方，加一行小字說明：
```html
<p style="font-size:.8125rem;color:var(--muted);margin-bottom:16px;">
  為什麼不是 2.0？你的路線在 1.0 涵蓋範圍內，升級後費用增加 $401，節省金額不變。
</p>
```
**成本：** 低（一行文字，可用靜態 copy 先佔位）

---

### M3｜結果卡缺少「了解 1.0 完整說明」連結
**來源：** H10
**影響：** 低～中 — 對於已接近決策但仍猶豫的用戶（Persona 1 的最後一哩路），結果卡是他們停留最久的地方。若他們想多了解一點，唯一出口是升級 2.0 的連結，沒有「我就想多了解 1.0」的路徑。
**修復：**
在升級連結旁加第二個選項：
```html
<div class="upgrade-link">
  已有 TPASS 1.0？<a href="#">查看升級 2.0 的費用差距 →</a>
  &nbsp;｜&nbsp;
  <a href="#plans" style="color:var(--ink-2)">了解 1.0 完整說明</a>
</div>
```
**成本：** 極低（改文字 + 加錨點）

---

### M4｜「已有 1.0，查升級」入口需有真實目的地
**來源：** H7
**影響：** 高（對 Persona 2 而言是核心路徑）— nav 右上角按鈕是整個設計對「現狀守衛」的主要入口。若只是捲動到首頁頂部或連到通用說明頁，等於入口失效。研究中 P2 和 P4 都明確說需要「你現在費用 → 升後費用 → 差多少」的精確試算，不是功能介紹。
**修復（需獨立設計）：**
為 Persona 2 設計一個獨立的「升級試算」流程，至少包含：
1. 確認目前 1.0 方案（選費率）
2. 輸入每月搭乘次數（或選路線）
3. 輸出：費用差距 + 「升級後現有設定是否保留」的明確保證

這屬於獨立頁面設計工作，無法靠 prototype 小修完成。
**成本：** 高（新流程，建議排入下一個設計迭代）
**當前緩解方案：** 將 nav 按鈕連結到頁面內的「升級說明」錨點，先有個去處，說明「你的 1.0 設定完整保留」，靜態版本比無目的地好。

---

### M5｜Step 1「← 上一步」在第一步應完全隱藏
**來源：** H1 Finding 2
**影響：** 低 — 目前以 `opacity: 0.35 + disabled` 處理，但按鈕形狀仍在，對部分用戶產生「是不是點不到？」的困惑。
**修復：**
```css
/* 在 JS goTo() 中，step 1 時隱藏返回按鈕 */
btnBack.style.visibility = currentStep === 1 ? 'hidden' : 'visible';
```
**成本：** 極低（一行 JS）

## 保留清單（不改，有意為之）

以下設計決策在評估中均表現良好，或已確認符合研究目標，**不應在後續迭代中輕易更動**。每條附「為什麼保留」的理由，供未來設計師參考。

---

### 保留 1｜兩欄英雄版面：左 value prop + 右 wizard
**保留理由：** V1 在 /design-shotgun 四個方向中脫穎而出，核心邏輯是「先看到承諾，馬上可以驗證」——左欄建立動機，右欄立刻提供行動路徑，無需捲動。這個結構直接回應研究中「網站沒辦法延續我的動力」的核心痛點。
**注意：** 保留概念，但左欄寬度比例（目前 960/480）可在 review [1] 中微調為 `max-width: 1200px + margin: 0 auto`，不影響核心結構。

---

### 保留 2｜Step 2 預設選取「5 天（週一至週五）」
**保留理由：** 唯一有預設值的步驟，有效防止空值錯誤（heuristic H5 肯定）。預設值覆蓋最大多數用戶情境（標準五日通勤族），不會誤導非主流用戶（他們自行換選即可）。

---

### 保留 3｜結果卡：大金額 → 路線說明 → 費用明細 → CTA 的層次
**保留理由：** 信息層次完全符合 DESIGN.md 原則一「數字優先」。$342 大字是決策觸發器，路線回顯建立信任（用戶確認這是算他的，不是範例），費用明細提供理性支撐，CTA 在信心建立完成後才出現。這個順序不應改變。

---

### 保留 4｜路線回顯（「竹北站 → 新竹站，每週 5 天」）
**保留理由：** 直接回應研究洞察——用戶需要確認「這個結果是算我的」。研究中 P1 多次強調她找不到「我的路線是否適用」的答案。結果回顯是對這個不信任的正面回應。

---

### 保留 5｜年化換算（「等於每年省 $4,104」）
**保留理由：** 符合行為設計中的「金額放大」原則——月省 $342 感覺有限，但年省 $4,104 接近半個月薪資，更能驅動決策。這是來自研究洞察（用戶需要清楚的 ROI 數字）的刻意設計。

---

### 保留 6｜資料更新日期標記（micro 文字）
**保留理由：** 研究中 P1 和 P4 都明確表達對「官方數字是否準確、是否過時」的疑慮（P4 找到 2023 年的 Medium 文章還在擔心資料是否仍有效）。日期標記是最低成本的信任訊號，務必在正式上線後持續更新，否則反效果。

---

### 保留 7｜Wizard hint text 的語氣
**保留理由：** 兩份評估文件都指出 hint text 品質高——「輸入你每天的起點和終點就好，轉乘站不用填」、「上下班來回都算一天」等語句自然、口語、主動消除預期困惑。這是 inline documentation 的最佳實踐，後續步驟調整時應維持相同語氣。

---

### 保留 8｜Nav「已有 1.0，查升級」的獨立 CTA 位置
**保留理由：** 對 Persona 2（現狀守衛）的入口設計邏輯正確——不和主要 CTA（新用戶試算）競爭注意力，而是作為次要路徑在 nav 靜靜等待。不應將它移入 hero 或與主流程混合，否則會稀釋兩個 persona 的各自清晰度。（目前問題是目的地，不是位置，見 M4。）

---

### 保留 9｜Trust strip 三欄：申辦免費 / 隨時終止 / 站點覆蓋
**保留理由：** 三項分別對應三個不同的決策顧慮：「辦了要花多少」「辦了能不能後悔」「辦了能不能用在我的路線」。信任訊號選擇精準，覆蓋三個主要放棄原因（研究中均有記錄）。不應增加第四項（資訊密度會超標），也不應替換現有三項為更通用的文字。

---

### 保留 10｜全站左對齊，無 center / left 混用
**保留理由：** review 的 AI slop 檢查確認此項。中英混排的介面尤其需要對齊一致性，中文字體在置中排列時易產生不自然的換行。全左對齊是已確認正確的決策。

## 實作備忘（給工程師，prototype 未實作但正式版需要）

以下來自 H9 N/A 條目與各修復建議的「進階版本」，屬於 prototype 無法模擬、需在真實實作時處理的行為：

| 項目 | 說明 | 優先 |
|------|------|------|
| 站名驗證 | Step 1 輸入的站名需對照資料庫，不在清單時顯示「找不到此站名，請確認是否為大眾運輸站點」，提供站名 autocomplete | 高 |
| 試算失敗降級 | API 失敗時不顯示空白，改為「無法即時計算，以下為該路線歷史平均值：$XXX」| 高 |
| 站名 autocomplete | 避免拼音輸入錯誤（如「竹北站」vs「竹北」vs「竹北車站」）影響試算結果 | 中 |
| 試算結果個人化 | Step 3 應根據 Step 1-2 的實際輸入動態計算，不顯示 hardcoded 的竹北→新竹數字 | 高（prototype 現為靜態展示） |
| 升級試算流程 | Persona 2 的「已有 1.0，查升級」需要獨立流程頁面（見 M4）| 高（下一迭代）|
| 資料日期自動更新 | 「資料更新：2026-05-01」應由 CMS 或 build pipeline 自動帶入，不能為靜態文字 | 中 |
